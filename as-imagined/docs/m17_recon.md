Investigation only. No implementation, no tests, no decisions.md/CLAUDE.md changes.
Purpose: give Rob and planning-Claude an accurate picture before locking M17a–M17x tier
boundaries, since M16 repeatedly found the original outline didn't survive contact with

Ground truth: `pokeemerald_expansion`. Methodology note up front — items 1–3 below are
fully source-verified (grepped, not assumed). Item 4's classification table leans on
established public knowledge of these abilities' mechanics, with source spot-checks only

## 1. Scope question — resolve before locking tier boundaries

**Open question for Rob.** CLAUDE.md/decisions.md's scope note says "All Emerald Expansion
moves and abilities... no Mega Evolution, Z-moves, or Dynamax," without saying explicitly
whether that means:

- (a) the full Gen 1–9 ability list `pokeemerald_expansion` ships (up to `ABILITIES_COUNT`
  = 318, including Paradox/Tera-adjacent/hack-custom abilities like `DRAGONIZE`,
  `EELEVATE`, `FIRE_MANE` that belong to non-Gen-3 or hack-custom species), or
- (b) only the abilities actually reachable by the 386-species Gen 3 roster this project
  targets (per the Project Scope note: "All 386 Gen III Pokémon... No Deoxys forms. No
  Unown").

**Finding that mostly resolves it in practice:** `data/pokemon.json` (M15's pipeline)
already assigns HIDDEN abilities from later generations to the 386-species roster (e.g.
Pidgey line has Tangled Feet [Gen 4] as its Hidden Ability, id 77; Wind Rider [Gen 9], id
274, is used by at least one roster species) — so the practical roster-relevant list is
**151 distinct ability IDs, ranging from id 1 to id 274**, NOT capped at "vanilla Gen 3
abilities only" (which would stop at id 76, Air Lock). This matches the project's
established practice of using GEN_LATEST config values elsewhere.

**What's still open:** should M17 (or a future milestone) also implement the ~167 abilities
that exist in `pokeemerald_expansion`'s full data but that NO species in the 386-roster can
ever have (Tera-gimmick abilities, Paradox-Pokémon-only abilities, hack-custom abilities,
and ordinary Gen 4+ abilities that happen not to land on any Gen-3 species's ability slots)?
Recommend scoping M17 to the **151 roster-relevant abilities only** and treating the rest as
explicitly out of scope (or a very-low-priority future bucket) — but this is Rob's call, not
assumed here.

## 3. Remaining — the actual diff

**151 roster-relevant − 12 implemented = 139 remaining.**

None of the outline's 13 named abilities (Overgrow/Blaze/Torrent/Swarm, Guts, Marvel Scale,
Hustle, Compound Eyes, Tangled Feet, Simple, Unaware, Trace, Forecast, Shed Skin, Bad
Dreams, Wonder Guard) are implemented — the outline's concern that some might already be
done incidentally did not pan out. However, two concrete issues WERE found while checking
this:

1. **Data pipeline bug (new finding):** `data/abilities/ability_0077.tres` (Tangled Feet)
   and `ability_0124.tres` (Pickpocket) are **missing entirely** — not "unimplemented,"
   genuinely absent as files. Root cause: `include/constants/abilities.h` defines these two
   IDs via a symbolic reference rather than a literal number
   (`ABILITY_TANGLED_FEET = ABILITIES_COUNT_GEN3`, `ABILITY_PICKPOCKET =
   ABILITIES_COUNT_GEN4`), and whatever script generated the 313 placeholder files evidently
   only parsed literal `= NNN,` patterns, silently skipping these two. Confirmed both IDs
   are genuinely used by roster species (Tangled Feet: Pidgey/Pidgeotto/Pidgeot/Doduo/
   Dodrio/Spinda; Pickpocket: Sneasel/Seedot/Nuzleaf/Shiftry) — this is a real gap that will
   need a data-file fix (two files, trivial) whenever either ability is implemented, not an
   M17 mechanic-design issue.
2. **Outline tier-placement accuracy, mixed:** the source-verified mechanic shapes below
   show the outline's Tier 1/Tier 2 split is directionally right but imprecise in both
   directions — see the "Outline vs. reality" callouts in the table below.

## 6. Infrastructure dependency flags — summary

Pulled out from the table above for visibility, per the task's explicit ask:

1. **Switch-out ability trigger hook — doesn't exist.** Needed by Regenerator (144) and
   Natural Cure (30). Only switch-IN hooks exist today (5 call sites, wired for M16d's
   hazards). Design once, use for both abilities.
2. **Transform mechanic — doesn't exist.** Needed by Imposter (150). No Transform move
   exists either. Likely deserves its own dedicated scoping pass, not a "Tier X" bucket.
3. **Trapping / switch-prevention check — doesn't exist.** Needed by Shadow Tag (23),
   Arena Trap (71), Magnet Pull (42). No "can this Pokémon voluntarily switch" gate exists
   in move/switch selection today.
4. **"Ability currently suppressed/ignored" plumbing — doesn't exist.** Needed by Mold
   Breaker (104) and Neutralizing Gas (256) together — design as one piece of shared
   infrastructure, not two separate features.
5. **Weather-change broadcast hook — doesn't exist.** Needed by Forecast (59). M11's
   weather system sets/expires weather but has no "notify all battlers a change occurred"
   event. Smaller lift than it looks, since the type-mutation itself can reuse the
   Conversion infrastructure directly.
6. **Item-consumption/loss event tracking — doesn't exist.** Needed by Unburden (84) and
   partially by Harvest (139, "last consumed berry" specifically). `ItemManager` currently
   has no concept of "this item was just removed/used up," only "does this mon currently
   hold item X."
7. **Per-individual gender — confirmed missing.** Needed by Rivalry (79).
   `PokemonSpecies.gender_ratio` exists (a probability), but `BattlePokemon` has no assigned-
   gender field at all (grepped both files directly) — a specific individual's gender is
   never rolled/stored anywhere in this codebase yet. Needs a new field before Rivalry can
   be scoped.
8. **New status type: infatuation/Attract — doesn't exist.** Needed by Cute Charm (56).
   Blocked on the Attract move itself not existing yet either.
9. **Move-flag additions needed (small, data-only):** `MoveData` doesn't yet have a "wind
   move" flag (Wind Rider, 274) or model weight-based move power (Heavy Metal/Light Metal,
   134/135) or trapping-move status. These are cheap (mirror the existing
   `sound_move`/`punching_move` flag pattern) but worth batching into whichever milestone's
   `gen_moves.py` pass touches them.
10. **Abilities with a currently-unimplemented move dependency, not an infra gap per se:**
    Oblivious (12, needs Attract+Taunt), Sticky Hold (60, needs an item-theft move), Cute
    Charm (56, needs Attract). These aren't blocked on new SYSTEMS, just on specific moves
    not existing yet — sequencing question, not a design question.
