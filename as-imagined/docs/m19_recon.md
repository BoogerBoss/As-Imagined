# M19 Pre-Recon — Full Move Scoping, Organized by Generation

Report-only recon. **No implementation code was touched in this session.** This
mirrors the recon-first, Rob-decides pattern already used for `docs/m17_recon.md`
(abilities) and `docs/m18_item_ledger.md` (items) — a complete inventory to review
before any M19 sub-tier gets scoped or sequenced.

Sources consulted: `data/moves.json` (935 entries incl. 1 dummy `MOVE_NONE`, this
project's own M15-pipeline reference dump — full stat/flag data, but genuinely NO
battle mechanics, confirmed by direct inspection of its own field list), `scripts/
gen_moves.py`'s `MOVES` dict (the actual current source of truth for "has real
`.tres` mechanics," counted directly rather than trusted from memory), and
`reference/pokeemerald_expansion/src/data/moves_info.h` (parsed directly for two
fields `data/moves.json` does NOT carry at all: `.multiHit`/`.strikeCount`, needed
to identify the multi-hit family precisely — a real gap in this project's own
existing data pipeline, flagged in Section D).

## Section A — Current baseline (confirmed directly, not from memory)

- **`data/moves.json`: 935 entries** (934 real moves + 1 dummy `MOVE_NONE`/"-" at id
  0, excluded from all counts below). This is the FULL reference catalog — every
  field pokeemerald-expansion's `moves_info.h` exposes in an already-flattened form
  (power/accuracy/pp/type/category/effect_name/all boolean flags/secondary_effect
  fields), but it is a **data dump only**. Confirmed by inspecting its own schema:
  there is no field here that executes anything — `effect_name` is a label, not
  code.
- **`scripts/gen_moves.py`'s `MOVES` dict: 91 entries.** This is the actual count of
  moves with real `.tres`-backed battle mechanics today, counted by direct regex
  extraction of `"id":` keys from the live script — not the ~91 figure carried over
  from memory, independently re-derived and confirmed identical.
- **Remainder: 934 − 91 = 843 moves** without real mechanics yet (one fewer than the
  task's own "~844" estimate — the discrepancy is exactly the dummy `MOVE_NONE`
  entry, which some prior informal count evidently included).
- Two-track architecture confirmed precisely: `data/moves.json` → (already exists,
  M15) full reference dump, no mechanics. `data/moves/move_NNNN.tres` (generated by
  `gen_moves.py`, loaded via `MoveRegistry`) → the 91 moves with real
  `BattleManager`/`DamageCalculator`/`StatusManager` logic behind them. Conflating
  "present in moves.json" with "has real mechanics" would have made every one of
  this recon's counts wrong — confirmed the distinction holds before doing anything
  else.

# Section C — Shared-mechanism opportunity clusters (ranked by leverage)

This is the highest-value section of this recon: each cluster below is one
unbuilt mechanism that would unlock multiple moves (and in three cases, items
and/or abilities already sitting deferred elsewhere in this project) at once —
the same "one mechanism, N unlocks" shape `[M18a]`'s type-boost dispatch and
`[M17n-6]`'s type-mutation mechanism already established for items/abilities.

## C1 — Multi-hit moves (31 moves) — highest leverage of any cluster found

**Confirmed via direct source parse** (`moves_info.h`'s `.multiHit`/`.strikeCount`
fields — NOT present in this project's own `data/moves.json` dump at all, a real
gap in the existing M15 pipeline, see Section D): 15 variable-hit moves (2–5 hits,
`.multiHit = TRUE`) — Double Slap, Comet Punch, Fury Attack, Pin Missile, Spike
Cannon, Barrage, Fury Swipes, Bone Rush, Arm Thrust, Bullet Seed, Icicle Spear,
Rock Blast, Tail Slap, Water Shuriken, Scale Shot — plus 16 fixed-hit moves
(`.strikeCount = N`) — Double Kick/Twineedle/Bonemerang/Double Hit/Dual
Chop/Gear Grind/Double Iron Bash/Dragon Darts/Dual Wingbeat/Twin Beam/Tachyon
Cutter (all 2-hit), Triple Kick/Triple Axel/Surging Strikes/Triple Dive (3-hit),
Population Bomb (10-hit, capped by accuracy-based reduced-hit-count logic in
source). **31 total.**

`MoveData.strike_count`/`MoveData.multi_hit` fields already exist in the schema
(confirmed dormant, never read by `BattleManager` — the exact gap `[M17n-5]`'s
own decisions.md entry flagged when deferring the Skill Link ability, and
`docs/m18_subtier_plan.md`'s Section C independently re-confirmed when deferring
Loaded Dice). Building the execution-loop mechanism (roll or read hit count,
repeat the damage-and-secondary-effect pipeline N times, handle a mid-sequence
miss/Substitute-break/target-faint correctly) is genuinely new work, but it is
the SINGLE highest-leverage mechanism in this whole recon:
- **Unlocks 31 moves directly** (10 with real mechanics already partially
  present as single-hit-only misresolutions today — these currently deal ONE
  hit's worth of damage where they should deal 2–10, a real, currently-live
  correctness gap, not just a missing feature).
- **Unblocks Skill Link** (ability, deferred at `[M17n-5]` for this exact reason
  — "always hit the max end of the multi-hit range").
- **Unblocks Loaded Dice** (item, deferred in `docs/m18_subtier_plan.md`'s
  Section C for this exact reason — becomes "a one-line max-end-of-roll
  modifier" once this mechanism exists).

## C2 — Binding-move family (11 moves) — second-highest leverage

**Confirmed via direct source parse** (`.additionalEffects[].moveEffect ==
MOVE_EFFECT_WRAP` or `MOVE_EFFECT_TRAP_BOTH`): Bind, Wrap, Fire Spin, Clamp,
Whirlpool, Sand Tomb, Magma Storm, Infestation, Jaw Lock, Snap Trap, Thunder
Cage. **11 total — a correction to `docs/m18_subtier_plan.md`'s own count**,
which listed only 8 when deferring Grip Claw (missing Jaw Lock, Snap Trap,
Thunder Cage — the three newest, Gen 7/8 binding moves). Traps the target for
4–5 turns (`B_BINDING_TURNS >= GEN_5` in this reference clone's config,
confirmed from source) of recurring end-of-turn damage, target cannot switch
out (with the usual Ghost-type/Shed Shell exceptions this project's trapping
infrastructure from `[M17f]` may already generalize — worth checking
`AbilityManager.is_trapped()`'s existing shape before building a parallel one).
- **Unlocks 11 moves** currently misresolving as plain single-hit damage with
  no trap/DoT component at all.
- **Unblocks Grip Claw** (item, deferred in `docs/m18_subtier_plan.md`'s Section
  C for this exact reason — becomes "a one-line 5-turn→7-turn duration
  modifier" once binding moves have a turn counter to extend).

## C3 — Terrain family (9 moves) — CROSS-REFERENCE: infrastructure already
## voided by a standing decision, not merely unbuilt

**Confirmed via source**: 4 terrain-setting moves (Electric Terrain/Grassy
Terrain/Misty Terrain/Psychic Terrain), 4 terrain-boosted-power moves
(Expanding Force/Misty Explosion/Rising Voltage/Psyblade), 1 terrain-type-

that decision or building a terrain system scoped ONLY to moves (no
terrain-ability interactions) — a genuinely different, narrower proposition
than what M17e voided, worth its own explicit go/no-go from Rob rather than
silently building or silently continuing to skip.

## C4 — Counter/Mirror-Move family (5 moves) — moderate leverage, likely cheap

Counter, Mirror Coat, Metal Burst, Comeuppance (`EFFECT_REFLECT_DAMAGE`, 4
moves — reflect 1.5–2× the last hit of a specific category taken this turn
back at its source) plus Mirror Move (`EFFECT_MIRROR_MOVE`, 1 move — reuse the
target's own last-used move against it). These share one underlying need:
tracking "the last move that hit this Pokémon this turn, and who used it" —
this project already tracks `_last_attacker`/`_last_attacker_move` for Destiny
Bond/Moxie/Aftermath purposes (`[M14b]`/`[M17n-8]`), so this cluster may be
CHEAPER than its "moderate" tier suggests — likely a small extension of
already-existing per-turn tracking rather than a new mechanism from scratch.
Worth a dedicated verification pass (not this recon) before final sizing.

## C5 — Weather-conditional heal family (4 moves) — moderate leverage, likely cheap

Moonlight, Morning Sun, Synthesis, Shore Up — each heals a DIFFERENT fraction
of max HP depending on current weather (typically 2/3 in sun, 1/4 in
rain/sand/hail, 1/2 in clear weather; Shore Up's own sand-specific bonus
differs slightly, confirm exact fractions from source before implementing).
This project already has a generalized weather system (`M11`) and a
generalized restore-HP move shape (`RESTORE_HP`, `[M16a]`) — this cluster is
likely a small extension combining two already-built systems rather than a
new mechanism, similar in shape to C4 above.

## C6 — Protect-family (10 further variants, base mechanism ALREADY BUILT)

Protect itself (move 182) is **already implemented** (one of the 91). The
other 10 `EFFECT_PROTECT` moves — Detect, Wide Guard, Quick Guard, Crafty
Shield, King's Shield, Spiky Shield, Baneful Bunker, Obstruct, Silk Trap,
Burning Bulwark (Max Guard, the 12th, is Dynamax-exclusive — see the
exclusion-candidate list) — each layer a small twist (side-wide instead of
single-target, a contact-punish effect, a stat-drop-on-contact effect, etc.)
on top of the SAME underlying "block the move, track the consecutive-use
success-chance falloff" mechanism this project already has fully working.
This is arguably not even "Tier 3 moderate" — closer to Tier 2, flagged
separately here only because each variant's own exact twist needs individual
source verification (per this project's own "don't assume symmetry between
similar-looking effects" discipline, repeatedly load-bearing across M17/M18).

## Clusters explicitly checked and found to be non-issues

- **OHKO-adjacent** (Guillotine/Horn Drill/Fissure/Sheer Cold): already fully
  implemented (`[M16a]`), all 4 accounted for in the "already implemented"
  baseline. No further work.
- **Weather-setting beyond M11**: Sandstorm/Rain Dance/Sunny Day/Hail/Snowscape
  (`EFFECT_WEATHER`, 5 moves) are a single well-precedented cluster, none
  implemented yet but trivially cheap — this project's weather system (`M11`)
  and weather-duration item mechanics (`[M18l]`-adjacent Rock items) already
  generalize directly; not flagged as needing a NEW mechanism.

   whether `BattlePokemon.ivs` (referenced in test helpers throughout this
   project, e.g. `bp.ivs = [0,0,0,0,0,0]`) is populated meaningfully outside
   test fixtures, or whether this is purely a test-scaffolding field with no
   real IV-generation system behind it for actual battle use — if the latter,
   Hidden Power's TYPE cannot be meaningfully determined for a real battle
   Pokémon yet, the same class of "no data to hook into" gap Section C's
   Attract/binding/multi-hit items already exemplify.

3. **Return (216) / Frustration (218) — need a friendship/happiness value.**
   Both `EFFECT_RETURN`-family and `EFFECT_FRUSTRATION` scale power directly
   with (or inversely with) the holder's friendship stat. Confirm whether this
   project tracks per-Pokémon friendship anywhere (`PokemonSpecies.
   base_friendship` exists per `data/pokemon.json`'s own schema — but that's a
   SPECIES base value, not a per-individual mutable stat that changes through
   play). If no mutable per-individual friendship exists, these two moves'
   power cannot be computed from real state — flag alongside Hidden Power as
   the same "missing per-individual stat" gap, not a mechanism problem.

4. **Terrain family (9 moves, Section C3)** — not a missing-data gap like the
   three above, but a standing PROJECT decision to re-confirm: terrain
   infrastructure was explicitly voided for abilities (`[M17e]`). Surfaced
   here again as a decision this recon found on the move side independently
   — needs the SAME explicit Rob confirmation before any of these 9 moves are
   scoped, not a silent inclusion or a silent continued exclusion either way.

5. **Nature-dependent moves**: none found. Checked directly — no move in this
   935-entry catalog has a nature-dependent effect (nature affects stat
   calculation globally, not any individual move's own behavior in this
   generation range). Confirmed absence rather than assumed; not a gap.

6. **EV-dependent moves**: none found, same direct-check method as natures
   above. No move's own effect reads EVs directly (EVs affect the underlying
   stat calculation, same as natures — no move-specific EV interaction exists
   in this move roster).
